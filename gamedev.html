I have experience with multiple engines, including Unity, Unreal Engine 4/5, and Godot, and published games on a variety of platforms.

TBDNE

Web-based Puzzle Game

This [Blank] Does Not Exist

Project Roles: Backend Programming, UI/UX Design
A 2-month project created as part of my 2023 internship with Retora Games. I worked as part of a team of 3 developers to create this Unity-based WebGL game. At the end of the internship, we launched the game at TBDNE.com and promoted it through various outlets.
TBDNE, Or "This [Blank] Does Not Exist", is a wordle-inspired puzzle game. Players are tasked with quickly differentiating between AI-generated images and real stock photos, with a score based on speed and accuracy.
Play it here!

Project Responsibilities

Database Design and Integration - I created a Firestore database with Google Firebase as a collection of player-agnostic game metrics, such as total scores and correct answer amounts. Using Firebase's JavaScript API, I was able to create C# functions that would call API methods and store game metrics.

UX Design - I designed and created a locally-stored "player history" page that would contain a player's overall average scores and score history. I referenced similar pages found on sites such as wordle and GuessTheGame, but adjusted the information displayed to better suit TBDNE's design. 

UI Design - I worked with Unity's UI Canvas system to create layouts, animations, and interactions for different sections of the game.

BitBoxers

Arcade Fighting Game

BitBoxers

Project Roles: Game Design Lead, Programmer
BitBoxers is a PvP game that blends concepts of fighting games with the auto-battler genre. It was programmed in C, and was playable over a local network with two separate ARM Microcontroller-based arcade devices. Players queue up 3 actions per turn from a group of 8 moves that have unique interactions with each other.

Game Trailer

Project Responsibilities

System Design - I conceptualized the core design of the game's combat system as well l as the interactions with specific physical controllers. We used separate input methods for offense and defense, and an extra analog input via a flex sensor for bonus damage.

Gameplay Programming - I was responsible for designing and programming the core gameplay loop and opponent AI system. I used a finite state machine system to control the info about moves and have them interact with each other.

Graphics Programming - I programmed the graphics drivers that displayed sprites and UI to the microcontroller's LCD screen. Due to the limited memory of the microcontroller, I created an indexing system for the colors of each pixel.
Move List and Move Interactions

Key Takeaways

• Paper prototypes are useful for conceptualizing combat systems with a small amount of discrete states. This combat design specifically was easy to test using cards and some notes to track health values.

• In-game feedback is important to convey the success or failure of certain actions within the game. We lacked specific sound effects or sprites that conveyed a successful hit, so players were often confused if their moves succeeded.

• If defensive options give no inherent reward (like punishing a move on block in a traditional fighter) they will be relatively unused. In our case, blocking had no reward besides trading a move slot for lowered damage, but there was no reason to do it as opposed to an attack.

OnlyGrabs

Unity Fighting Game

OnlyGrabs

Solo-Developed Project
OnlyGrabs is a prototype of a wrestling-themed fighting game that I created in Unity using C# and the Universal Fighting Game Engine. This was my first exploration into fighting game development, and within two weeks I was able to design and implement the moveset of singular character, including movement, frame data, and damage amounts.

Gameplay Video

Play on itch.io!

Project Devlog

This was a solo project created as a final for my 'Game Prototyping' course. My initial idea was for a fighting game focused specifically around grabs, without strikes of any type. To minimize the amount of technical work and focus mostly on design, I made use of a fighting game framework for Unity.
In traditional fighting games, throws complete the rock-paper-scissors system, defeating an opponent's blocking defense against strikes.

In the absence of strikes, there was no need for blocking either, so I instead devised a system that would revolve around grabs and dodges.

Players would move back and forth to bait and punish grab attempts with a dodge or roll.
Player moveset concept
UFE2 hitbox editor
Using the UFE move editor I created the move's hitboxes and hurtboxes. It also allowed me to modify the startup and recovery frames, as well as the knockback.

An interesting quirk of the system was that throughout the duration of the throw, an opposing throw would 'tech' it to cancel it out. I created a unique throw whiff system to allow missed throws to be punished.
When testing and adjusting the moves in the game, I found it helpful to keep track of their frame data in a table, similar to the information tables found on fighting game wikis like Dustloop or Mizuumi.

I also kept track of previous iterations of certain moves and notes on their specific requirements
Frame data table

Lessons Learned

• Games that rely on precise spacing and limited movement need accurate hitboxes to make combat feel "fair" 

• Movement options that are intended to be 'unreactable' have to take into account travel time as well as frame data

• Even if grabs and throws are thematically important to a wrestling theme, it's more representative of the grappler playstyle to create mixups between strikes and throws and go for hard reads with big rewards

RushRushVr

VR Microgame Collection

RushRushVR

Project Roles: Game Designer, Gameplay Programmer
RushRushVR is a collection of VR Microgames, inspired by games such as WarioWare and Rhythm Heaven. This was a 6-week project created in Unity using C# and the XR Interaction toolkit. Working with a team of 4 other developers, we were each responsible for 2 specific microgames in their entirety, including design, asset creation, and implementation.

Gameplay Video

Download on itch.io!

Project Responsibilities

System Design - I was responsible for the overall core design of the game and it's general mechanics. I recognized during a brainstorming session that there was an open niche for this type of game in the VR space, and that arcade-type score challenges are a good fit for VR.

Level Design   and Gameplay Programming - I created the "Bug Catcher" microgame, using a simple collection of interactions and combining them with a custom behavior pattern created for the bugs' movements. 

Level Design and Gameplay Programming - I created the "Ice Cream Match" microgame, iterating on a very simple interaction by adding pattern recognition as an inherent mechanic to change the gameplay.
Bug Catcher Microgame
Ice Cream Match Microgame

Feedback and Design Changes

• The ice cream microgame originally had the "target cone" in the left hand and the sequence of flavors falling from the ceiling was fully random. Based on playtesting and feedback about clarity, I made the target cone more clear and made the falling scoops pseudo-random, to guarantee that the player would not lose from RNG.

• The bugs in the bug catcher microgame originally flew in a fully random, erratic pattern that would sometimes draw them too far from the player. To resolve this, I adjusted their pattern to follow a sin wave while circling the player, randomizing the directions, speed, and starting positions to retain some uniqueness on each attempt.

Lab Echo

VR Narrative Puzzle Game

Lab Echo

Project Roles: Game Design Lead, Gameplay Programmer
Lab Echo is a narrative puzzle game that is playable on Windows, Mac, and Oculus Quest 2. It was created as a semester-long project with a team of 7 developers, using Unreal Engine 5 and a combination of C++ and Blueprint scripting. I was the game design lead for the project, responsible for system design, level design, and the game's narrative.

Game Trailer

Download on itch.io!

Project Responsibilities

System Design - I designed the core gameplay systems of the game, taking inspiration from titles such as Return of the Obra Dinn and Case of the Golden Idol  to create puzzles based around exploration and deductive reasoning. I also designed and programmed a secondary puzzle system inspired by binary sudoku puzzles to serve as a gap between narrative segments.

Level Design - I conceptualized 3 connected level layouts that used environmental storytelling to create a puzzle for the player. By reverse-engineering the order of events and using locations of props as context, the player could solve each narrative puzzle.

Narrative Design  - I wrote the dialogue and overarching plot of the game, breaking down different segments of the narrative into scenes that could be explored in puzzles to gradually understand the mystery.

Organizing Playtests  - I organized and collected feedback from weekly playtests to create improvements to player control, UI, puzzle structure, and narrative.
First Level Layout and Associated Props

Feedback and Lessons Learned

• Designers and Gameplay Programmers should communicate very often to ensure that crucial features for different stages of development are being prioritized properly. We ran into miscommunications after certain design changes where unimportant or removed mechanics were still being worked on by programmers.

• Vertical slice prototype should include a basic version of all features to ensure that the gameplay flow can be properly understood and limit scope creep later.

• Sometimes the feedback from playtests comes from a place of unfamiliarity with genre norms, especially for niche genres. It's the job of the designer to understand the underlying issue behind the feedback and create a solution, rather than applying a band-aid to the complaint.

The Bow Game

VR Archery Simulator

The Bow Game

Project Roles: Gameplay Programmer
The Bow Game is a VR Archery Game prototype that was created with a team of 5 developers over the course of 3 weeks. The goal of the project was to get accustomed to creating VR scenes and interactions in Unity, as well as understand the XR Interaction Toolkit and API.

Gameplay Video

Project Responsibilities

Gameplay Programming - My contribution was mostly focused around creating the enemies and programming their movement patterns. I created a bezier curve system that would define a path for each enemy, and had a series of variations for each prefab to create unique patterns.

VR UI Programming - I also created the implementation of the start and ending screens for the VR Game. The VR interactions were similar to using the Unity UI canvas for 2D and 3D games, so it didn't require much adjustment.
